# Background and Finished Tasks Design

## Goal

Show Claude Code-style background and finished tasks in Claude Remote Web, covering both running sessions and tool-level work inside sessions. Users should be able to see what is still running, what completed, what failed or was interrupted, and jump back to the relevant session context.

## Scope

Implement a task view derived from existing append-only session events. Do not replace the event log, mutate historical events, or add a second source of truth for process state.

The first implementation targets tool-level tasks for Claude events that expose stable identifiers or recognizable start/result pairs, plus session-exit cleanup for incomplete tasks.

## Architecture

Add a backend task projection layer that reconstructs task state by reading persisted session metadata and event logs. The projection is ephemeral and rebuildable: event logs remain authoritative, and task lists are computed from them when requested.

A task has:

- Stable task id, preferably from tool call id, shell id, task id, or agent id when present.
- Owning session id.
- Tool kind, such as Bash, Agent, Workflow, or unknown tool.
- Human-readable title or summary.
- Status: `background`, `completed`, `failed`, or `interrupted`.
- Start and end timestamps when derivable.
- References to the event ids or event indexes that created and completed the task.
- Short result summary for finished tasks.

The projection layer belongs on the backend so the frontend does not need to duplicate Claude event parsing rules.

## Event Interpretation

Task extraction should be conservative. If the system cannot identify a clear task start, stable id, or completion relationship, it leaves the data as normal events instead of creating a guessed task.

Initial rules:

1. A recognized tool-use event creates or updates a task in `background` state.
2. A matching tool-result or completion event marks the task as `completed` or `failed`, based on the event payload.
3. A session process exit marks any still-background task in that session as `interrupted`.
4. Failed sessions mark still-background tasks as `interrupted`, not `failed`, unless the task itself has an explicit failure result.
5. Unknown tool names are allowed, but only when a stable id exists.

This preserves full raw payloads and keeps task rendering additive.

## API

Add two read-only endpoints:

```text
GET /api/tasks
GET /api/sessions/:id/tasks
```

`GET /api/tasks` returns tasks across all sessions, grouped or sortable by status for the global Tasks panel. `GET /api/sessions/:id/tasks` returns tasks for one session for the in-session panel.

The response shape should be explicit and frontend-friendly:

```json
{
  "background": [],
  "finished": []
}
```

Finished includes `completed`, `failed`, and `interrupted` tasks. Each task carries enough session and event reference data for the frontend to navigate to the source conversation.

## Frontend Experience

Add a Tasks area with two groups:

- Background tasks: currently running tool tasks, showing tool name, summary, session, and elapsed time.
- Finished tasks: completed, failed, and interrupted tasks, showing final status, finished time, and a short result summary.

Clicking a task selects its owning session and scrolls or highlights the related event when possible. If precise event positioning is not available yet, selecting the session is still required and event positioning can be added with the event reference model.

The active session view also includes a compact task list for that session, so users can monitor work without switching away from the conversation.

## Refresh Model

The frontend fetches task lists on load and refreshes them after receiving relevant WebSocket session events. A lightweight polling fallback is acceptable if the current WebSocket wiring does not cover all sessions globally, but task state should still be generated by the backend projection endpoint.

## Error Handling

Task projection errors should not break the session list or conversation view. If one event cannot be parsed as a task, skip task creation for that event and keep rendering the raw event normally. If a task endpoint fails, the frontend shows a small non-blocking error in the Tasks area and leaves the rest of the app usable.

## Testing

Backend tests should cover:

- Tool-use event creates a background task.
- Matching tool-result marks a task completed.
- Explicit tool failure marks a task failed.
- Session exit interrupts unmatched background tasks.
- Unknown or ambiguous events do not create guessed tasks.
- `/api/tasks` returns tasks across sessions.
- `/api/sessions/:id/tasks` returns only that session's tasks.

Frontend tests should cover:

- Background and finished grouping.
- Status rendering for completed, failed, and interrupted tasks.
- Clicking a task selects the owning session.
- Endpoint failure displays a non-blocking task-panel error.

Manual verification should run the app, create a session, trigger a long-running background tool task, observe it in Background tasks, then confirm it moves to Finished tasks after completion or interruption.
